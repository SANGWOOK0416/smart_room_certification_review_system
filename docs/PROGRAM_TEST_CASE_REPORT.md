# 자취방 거래 안심리뷰 시스템 테스트 케이스 문서

## 1. 문서 개요

본 문서는 **자취방 거래 안심리뷰 시스템**의 전체 기능을 대상으로 한 테스트 케이스를 정리한 문서이다.

테스트 범위는 특정 백엔드 API에 한정하지 않고, 다음 영역 전체를 포함한다.

- 사용자 기능: 회원가입, 로그인, 지도 탐색, 매물 상세 조회, 리뷰 등록, 마이페이지
- 관리자 기능: 리뷰 승인/반려/삭제, 사용자 관리, 전국 실거래가 수집
- 지도 기능: Kakao Map 연동, 검색, 마커 표시, 매물 상세 패널
- 리뷰 검증 기능: 계약서 업로드, 관리자 승인, 승인 리뷰 공개
- 공공데이터 기능: 국토교통부 실거래가 API 호출 및 DB 저장
- 월세 적정성 계산기: 환산월세, 면적 보정, IQR 기반 이상치 판단
- DB 기능: User, Building, Transaction, Review, VerificationDocument 관계 검증

## 3. 국토교통부 API 호출 파라미터

본 프로젝트는 국토교통부 실거래가 API를 호출할 때 다음 파라미터를 사용한다.

| 파라미터 | 의미 | 예시 |
|---|---|---|
| `serviceKey` | 국토교통부 API 인증키 | 환경변수 `MOLIT_SERVICE_KEY` |
| `LAWD_CD` | 법정동 앞 5자리 지역 코드 | `47130` |
| `DEAL_YMD` | 거래 조회 연월 | `202605` |
| `pageNo` | 조회 페이지 번호 | `1` |
| `numOfRows` | 한 번에 가져올 데이터 수 | `100` |
| `kind` | 호출할 API 종류 | `aptRent`, `singleHouseRent` |

### API 종류

| kind | 수집 대상 |
|---|---|
| `aptTrade` | 아파트 매매 |
| `aptRent` | 아파트 전월세 |
| `officetelRent` | 오피스텔 전월세 |
| `rowHouseRent` | 연립/다세대 전월세 |
| `singleHouseRent` | 단독/다가구 전월세 |

## 4. 블랙박스 테스트 케이스

블랙박스 테스트는 내부 구현을 보지 않고, 사용자가 실제 화면에서 수행하는 동작을 기준으로 검증한다.

| ID | 구분 | 테스트 항목 | 테스트 절차 | 기대 결과 | 근거 및 타당성 |
|---|---|---|---|---|---|

| BB-02 | 공통 | 사용자 로그인 | 등록된 사용자 계정으로 로그인한다. | 마이페이지와 리뷰 등록 기능을 사용할 수 있다. | 인증 상태에 따라 사용 가능한 기능이 달라지므로 필수 테스트이다. |
| BB-03 | 공통 | 로그아웃 | 로그인 상태에서 로그아웃 버튼을 클릭한다. | 세션이 제거되고 리뷰 등록이 제한된다. | 사용자 상태 전환이 정상적으로 처리되는지 확인한다. |
| BB-04 | 메인 UI | 초기 화면 로딩 | 사이트에 접속한다. | 지도, 검색창, 필터, 매물 마커가 표시된다. | 지도 기반 서비스의 첫 화면이 정상 동작해야 한다. |

| BB-07 | 검색 | 도로명주소 검색 | `북문로55번길 15-1` 같은 도로명주소를 입력한다. | 해당 주소 주변으로 지도가 이동하거나 매물이 표시된다. | 주소 기반 리뷰 등록 매물을 찾기 위한 핵심 기능이다. |

| BB-10 | 지도 | 전세/월세 마커 구분 | 전세 매물과 월세 매물을 각각 확인한다. | 전세와 월세가 시각적으로 구분된다. | 계약유형을 직관적으로 구분해야 한다. |
| BB-11 | 필터 | 월세 필터 | 월세 필터를 클릭한다. | 월세 매물만 표시된다. | 전세와 월세가 섞이면 비교가 불가능하다. |
| BB-12 | 필터 | 전세 필터 | 전세 필터를 클릭한다. | 전세 매물만 표시된다. | 계약유형 필터 정확성 검증이 필요하다. |
| BB-13 | 필터 | 아파트 분류 | 아파트 매물 표시 상태를 확인한다. | 아파트가 별도 유형으로 표시된다. | 원룸, 오피스텔, 아파트 구분 요구사항을 검증한다. |

## 5. 화이트박스 테스트 케이스

화이트박스 테스트는 내부 코드, DB 구조, API 흐름, 계산식을 기준으로 검증한다.

| ID | 구분 | 테스트 항목 | 내부 검증 대상 | 기대 결과 | 근거 및 타당성 |
|---|---|---|---|---|---|
| WB-01 | 프론트 | 검색어 상태 관리 | 검색 input state, API 호출 파라미터 | 검색어가 백엔드 `keyword`로 전달된다. | 검색 결과와 API 결과가 일치해야 한다. |
| WB-02 | 프론트 | 필터 상태 관리 | 월세, 전세, 아파트 필터 state | 선택한 필터에 맞는 매물만 렌더링된다. | 화면 필터와 데이터 필터 불일치 방지이다. |
| WB-03 | 프론트 | 지도 중심 이동 | Kakao Map center, level 변경 로직 | 검색/지역 선택 시 지도 중심이 변경된다. | 사용자 검색 위치로 이동해야 한다. |
| WB-04 | 프론트 | 마커 렌더링 | 매물 배열을 마커로 변환하는 로직 | 매물 수와 병합 기준에 맞게 마커가 표시된다. | 지도 표시 오류 방지이다. |
| WB-05 | 프론트 | 상세 패널 상태 | 선택된 building state | 마커 클릭 시 해당 매물 정보가 표시된다. | 다른 매물 정보가 뜨는 오류를 방지한다. |
| WB-06 | 프론트 | 리뷰 탭 렌더링 | `/api/reviews?buildingId=` 응답 처리 | 승인 리뷰 목록이 상세 탭에 표시된다. | 리뷰 수만 보이고 내용이 안 보이는 문제를 방지한다. |
| WB-07 | 프론트 | 거래 그래프 렌더링 | Transaction 배열을 차트 데이터로 변환 | 날짜와 월세 데이터가 그래프에 표시된다. | 실거래 추이 시각화 검증이다. |
| WB-08 | 프론트 | 계약서 미리보기 | fileUrl, mimeType 분기 | 이미지와 PDF가 각각 열람된다. | 파일 형식별 UI 처리 검증이다. |

## 6. 테스트 설계 근거

### 6.1 프로그램 전체 흐름 기준

본 프로그램은 단일 기능이 아니라 다음 흐름이 연결된 서비스이다.

```text
회원가입/로그인
→ 지도에서 매물 탐색
→ 실거래 정보 확인
→ 계약서 첨부 리뷰 등록
→ 관리자 승인/반려
→ 승인 리뷰가 지도와 매물 상세에 반영
→ 월세 계산기로 시세 적정성 판단
```

따라서 특정 API 하나만 테스트하는 것은 충분하지 않다.  
사용자 화면, 관리자 화면, 지도, DB, 외부 API, 계산 로직을 모두 테스트해야 프로그램 전체의 안정성을 확인할 수 있다.

### 6.2 블랙박스 테스트의 필요성

블랙박스 테스트는 실제 사용자가 겪는 문제를 찾기 위한 테스트이다.

사용자는 내부적으로 `Review`, `Building`, `Transaction`이 어떻게 연결되는지 알지 못한다.  
사용자는 다음 결과만 보고 프로그램이 정상인지 판단한다.

- 리뷰를 등록했을 때 관리자 승인 대기로 들어가는가?
- 승인된 리뷰가 지도와 매물 상세에 보이는가?
- 같은 매물이 여러 마커로 나뉘지 않는가?
- 전세 버튼을 눌렀을 때 전세만 보이는가?
- 계약서 파일이 열리는가?
- 월세 계산 결과가 이해 가능하게 표시되는가?

따라서 화면 기능과 사용자 흐름 중심의 블랙박스 테스트가 필요하다.

### 6.3 화이트박스 테스트의 필요성

화이트박스 테스트는 화면 뒤쪽의 내부 로직을 검증하기 위한 테스트이다.

이 프로젝트는 내부 로직의 정확성이 결과 신뢰도에 직접 영향을 준다.

- 국토교통부 API 파라미터가 잘못되면 실거래 데이터가 수집되지 않는다.
- 주소 정규화가 잘못되면 같은 매물이 여러 마커로 표시된다.
- 리뷰 승인 시 거래화 로직이 동작하지 않으면 거래 수가 올라가지 않는다.
- 환산월세 공식이 틀리면 월세 적정성 계산 결과가 왜곡된다.
- IQR 상한선 계산이 틀리면 정상 매물을 고월세로 잘못 판단할 수 있다.

따라서 내부 코드, DB 관계, API 조건, 계산식을 검증하는 화이트박스 테스트가 필요하다.

## 8. 결론

본 테스트 케이스는 백엔드 API만을 대상으로 한 것이 아니라, **프로그램 전체 기능**을 대상으로 설계되었다.

블랙박스 테스트를 통해 사용자가 실제 화면에서 경험하는 기능을 검증하고, 화이트박스 테스트를 통해 내부 로직과 데이터 흐름의 정확성을 검증한다.

특히 본 프로젝트의 핵심인 지도 기반 매물 탐색, 계약서 기반 리뷰 검증, 승인 리뷰의 지도 반영, 국토교통부 실거래가 수집, 월세 적정성 계산 기능은 서로 연결되어 있기 때문에 두 방식의 테스트를 함께 수행하는 것이 타당하다.

발표용 요약 문장은 다음과 같다.

> 본 프로젝트는 지도 탐색, 리뷰 검증, 공공데이터 수집, 월세 적정성 계산 기능이 서로 연결된 시스템이므로, 사용자 관점의 블랙박스 테스트와 내부 로직 중심의 화이트박스 테스트를 함께 설계하였다. 이를 통해 화면 기능의 정상 동작뿐 아니라 리뷰 병합, 거래 수 반영, 환산월세 및 IQR 계산의 정확성까지 검증할 수 있다.
